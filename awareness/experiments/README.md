# Experiments

Standalone helpers, demos, and one-off preview generators that are NOT part of the runtime app. None of these are referenced by `index.html`, `preview.html`, etc., or by any `npm` script.

## Files

| File | Purpose |
|---|---|
| `qr-demo.html` | Standalone QR-code rendering demo. Open directly in a browser. |
| `gen_previews.js` | Node script that loads the newsletter builder and emits static HTML previews (`corp_preview2.html`, `test_preview.html`) to this folder. |
| `corp_preview2.html` | Generated by `gen_previews.js` — Corporate Alert template preview with example content. |
| `test_preview.html` | Generated by `gen_previews.js` — Security Dispatch (Test) template preview. |
| `email_audit.js` | Node script that reads `corp_preview2.html` and audits it for email-safety issues (size, tag counts, etc.). Depends on `gen_previews.js` running first. |

## How to run

**From the project root** (paths inside the scripts are relative to CWD):

```bash
node experiments/gen_previews.js   # regenerates corp_preview2.html + test_preview.html
node experiments/email_audit.js    # audits corp_preview2.html
```

## Known staleness

`gen_previews.js` was written before the Phase 3 split. It loads `js/newsletter_builder.js` only and expects `TEMPLATE_BUILDERS` to be pre-populated. After the split, templates self-register from `js/newsletter/bank_page.js` and `js/newsletter/core_templates.js`. If `gen_previews.js` errors with "template not registered" or similar, add these two `vm.runInContext(fs.readFileSync(...), context)` calls after the `newsletter_builder.js` load:

```js
vm.runInContext(fs.readFileSync('js/newsletter/bank_page.js', 'utf8'), context);
vm.runInContext(fs.readFileSync('js/newsletter/core_templates.js', 'utf8'), context);
```

The sibling files reference `window.App.NewsletterBuilder`; `gen_previews.js` already sets `context.window = context` so this should work without further changes.

## Why these are not deleted

They were useful for one-off design/QA work and may be useful again. They're kept out of the project root so the root reflects only the production app surface (9 HTML entrypoints + config files).
