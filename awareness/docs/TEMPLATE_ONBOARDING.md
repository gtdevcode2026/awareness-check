# Template Onboarding Playbook

How to add a new newsletter template to this codebase. Reusable across future onboardings — read this first, then execute the recipe.

## Architecture Reality (Read This First)

The most common misconception: **templates are NOT loaded from disk at runtime.** The HTML files under `templates/imported-standalone/` are **visual reference documents** only. At runtime, every template is a JavaScript function that returns an HTML string with `{{TOKEN}}` placeholders, defined inline inside `js/newsletter_builder.js`.

Key locations in `js/newsletter_builder.js`:

- `escapeHtml(s)` helper at line 36-37 — sole substitution-escape function. Reuse, do not redeclare.
- `TEMPLATE_CATALOG` array at line 131 onwards — the catalog of every template, read by the UI to populate the picker grid on `index.html`.
- `getTemplateCatalog()` at the bottom — returns the catalog plus accessibility metadata. No change needed when adding new templates; new rows are picked up automatically.
- `build(format, cfg, arts, opts)` switch — routes the template `id` to its build function.
- `applyRenderProfile()` — wraps every build function's output with `data-template-id`, `data-template-name`, `data-render-channel` attributes. No change needed.

The email-safe folder at `templates/imported-email-safe/` is generated by `scripts/normalize-imported-templates.mjs`. **Do not hand-edit it.** Run `npm run templates:normalize-imports` after adding files to `imported-standalone/` if you want a side-by-side QA artifact.

## The Four Patch Points

Any new template — variant or brand-new — needs exactly four edits, plus a test update.

1. **`TEMPLATE_CATALOG` row** in `js/newsletter_builder.js` (near line 131). Shape:
   ```js
   { id: 'your_id', name: 'Visible Name', tags: ['tag1', 'tag2'], channels: ['email-safe', 'screen-safe'], visualProfile: 'phishing-brief', recommended: 'One-line description shown under the card.' },
   ```
   - `id` is used in the `build()` switch and emitted as `data-template-id`. Underscores are valid; existing IDs use no separators by convention.
   - `name` is the picker card title.
   - `channels` is a subset of `['email-safe','print-safe','screen-safe']`.
   - `visualProfile` is a free-form identifier — pick one of the existing values for affinity or invent a new one.

2. **Build function** in `js/newsletter_builder.js`, inserted alongside related templates. Signature, defaults, helpers, token-build block, and substitution loop must mirror an existing function (e.g. `buildPhishingBrief` at line 2442). Only the `HTML` template literal differs.

3. **`build()` switch case** (near the bottom of `newsletter_builder.js`):
   ```js
   case 'your_id': html = buildYourFunction(cfg,arts,wo,lk,p,qr,il); break;
   ```

4. **`CONTEXT.md`** — extend the existing `js/newsletter_builder.js` bullet (around line 32) with a note about your template's token contract if it differs from the phishing-brief 11-token contract. No new docs file required.

5. **`tests/unit/app-modules.test.js`** — two assertions enumerate `catalog.length` (around lines 211 and 358). Bump both by 1 when adding a template (decrement when removing).

## The 11-Token Phishing-Brief Contract

The phishing-brief layout family (`phishingbrief`, `bankpage1_static`, future siblings) shares this contract. AI-generated content is sourced from the `arts[]` array passed into the build function:

| Token | Source |
|---|---|
| `INTRO` | `firstSentence(summaryOf(arts[0]))` → `titleOf(arts[0])` → default |
| `SECTION1_BULLET1..4` | `titleOf(arts[1..4])` → `firstSentence(summaryOf(...))` → default |
| `SECTION2_BULLET1..3` | `titleOf(arts[5..7])` → fallback chain → default |
| `SECTION3_BULLET1..3` | `titleOf(arts[8..10])` → fallback chain → default |

Substitution loop (used by every template that adopts this pattern):

```js
let out = HTML;
for (const k of Object.keys(tokens)) {
  out = out.split('{{' + k + '}}').join(escapeHtml(tokens[k]));
}
return out;
```

**Every token is HTML-escaped.** Do not introduce per-token bypass lists — if a token must contain markup, design a different token format or pre-escape only the markup you allow.

If your template needs a different token set, define it in the function and document it in the `CONTEXT.md` extension for that template.

## Two Onboarding Paths

### Path A: Variant of an Existing Template

Use when you have a hand-edited copy of an existing layout (e.g., a `…back.html` working file) that you want as a sibling template.

1. Verify the base64 image payloads match between your HTML file and the existing JS template literal (`PowerShell: (Get-Content $file)[lineN].Length` — base64 image lines should match byte-for-byte). If they do, you can duplicate the existing JS function and apply surgical text diffs.
2. Identify the structural diffs between your HTML and the original. List each as a precise `old → new` string pair.
3. Duplicate the existing function and apply the diffs. Easiest mechanically via a one-shot PowerShell script: extract the function span, rename, apply each diff via `.Replace()`, splice back in. **Delete the script after the change lands** — it is single-use noise.

A worked example exists in git history: `bankpage1_static` was added this way as a sibling of `phishingbrief`.

### Path B: Brand-New Template

Use when you have a fresh HTML fragment with no existing JS counterpart.

1. Place the file in `templates/imported-standalone/`. Confirm it has the wrapper `<div data-nl-import-body="1">`. Do not reformat or rewrap — base64 data URIs break if line-split.
2. Identify your token placeholders. Either reuse the 11-token contract (recommended) or define a new one. Insert `{{TOKEN}}` markers in the HTML at the right spots.
3. In `newsletter_builder.js`, write a new function that:
   - Stores the HTML as a JS template literal (escape any literal backticks or `${` — the phishing-brief template has none, your template may).
   - Builds a `tokens` object from `arts[]` using `titleOf` / `summaryOf` / `firstSentence` helpers (already in scope inside any build function via closure).
   - Defines `defaults` for every token so empty `arts[]` still renders.
   - Runs the standard substitution loop with `escapeHtml`.
4. Wire up the catalog row + switch case + test count bump.

## Verification Gate

Run in this order. Stop and triage if any step fails — do not paper over failures.

1. **Lint:** `npm run lint` — must show 0 errors. Pre-existing warnings (about 114 `no-unused-vars` on catch parameters) are noise; ignore them, do not chase.
2. **Unit:** `npm run test:unit` — all tests pass. If you forgot to bump the catalog count, two tests will fail with `21 !== 22`-style messages. Fix and re-run.
3. **Programmatic render:** confirm the new template renders with 11 fixture articles, produces no real token leakage, and stays under 102 KB (Gmail clipping threshold). One-liner:

   ```bash
   node -e "
   const vm = require('vm'), fs = require('fs');
   const ctx = { window:{}, localStorage:{getItem:()=>null,setItem:()=>{},removeItem:()=>{},clear:()=>{}}, URL, Date, console, setTimeout, clearTimeout, AbortController, navigator:{hardwareConcurrency:4}, performance:{now:()=>0} };
   ctx.window = ctx;
   ctx.App = { Utils: { log(){}, fmtDate(v){return v||'';}, stripTags(v){return String(v||'').replace(/<[^>]*>/g,'');}, truncate(v,l){return String(v||'').slice(0,l);}, normalizeWebUrl(v){return String(v||'');}, escapeHtml(v){return String(v||'');} } };
   vm.createContext(ctx);
   vm.runInContext(fs.readFileSync('js/newsletter_builder.js','utf8'), ctx);
   const arts = Array.from({length:11},(_,i)=>({title:'Test '+(i+1),summary:'Summary '+(i+1)+'.'}));
   const html = ctx.App.NewsletterBuilder.build('YOUR_ID_HERE', {nlTitle:'T',nlKicker:'K'}, arts, {});
   const leak = (html.replace(/<!--[\s\S]*?-->/g,'').match(/\{\{[A-Z_0-9]+\}\}/g) || []);
   console.log('Size:', html.length, 'Leakage:', leak.length, leak);
   "
   ```

   Replace `YOUR_ID_HERE` with your template id. Expect: size under 102000, leakage 0.
4. **Regression:** render the templates that previously existed using the same fixture, compare to a baseline captured before your change. Output should be byte-identical for those.
5. **Browser:** `npm run serve` → open `http://127.0.0.1:4173/index.html`. Confirm the new card appears in the picker grid. Click PREVIEW with articles selected. View source to inspect.
6. **Full gate:** `npm run verify` (lint + unit + e2e + baseline audit) — must be green before committing.

## Pitfalls We've Already Hit

- **PowerShell 5.1 reads files as Latin-1 unless a UTF-8 BOM is present.** If you write a transform script containing non-ASCII characters (em-dashes appear frequently in the templates' HTML comments), the script must be saved with BOM, else `Contains()` / `Replace()` calls silently mismatch. Use `[System.IO.File]::WriteAllText(path, text, [System.Text.UTF8Encoding]::new($true))` to add the BOM, or use plain ASCII in your script.

- **Base64 image lines blow past the Read tool's 25K-token cap when read in ranges.** In `Template 13 - Phishing Brief (ABC Company)…html` the image lines are at 56 / 88 / 108 with byte counts ~32KB / ~45KB / ~3KB. Read each line in isolation or skip past them. The JS template literal copies these lines verbatim, so they exist at the same byte counts inside `newsletter_builder.js` too.

- **`{{TOKEN}}` strings appear inside HTML doc-comments.** Every function's leading template literal includes a documentation block (`<!-- TEMPLATE: ... AI-FILLED PARTS (replace these {{TOKEN}} strings each edition): -->`). After substitution, those literal `{{TOKEN}}` and `{{TOKEN_NAME}}` strings remain in the rendered HTML *inside the comment*. Token-leakage checks must `.replace(/<!--[\s\S]*?-->/g, '')` before grepping for `{{`, or they'll false-positive.

- **The two catalog-count test assertions are easy to forget.** `tests/unit/app-modules.test.js` enumerates `catalog.length` twice (around lines 211 and 358) with hard-coded numbers. Both must be bumped on each catalog change.

- **The standalone HTML file's name has no runtime effect.** Trailing suffixes like `…back.html` are tolerated by `scripts/normalize-imported-templates.mjs` (which auto-discovers all `*.html` in the folder). Rename freely for human clarity if you want; nothing in JS depends on the filename.

- **`white-space:nowrap` on hero headlines clips on mobile.** Email clients render at viewport widths as narrow as 320px. A nowrap heading at 38px Arial Bold requires ~430px and will horizontally overflow on phones. Use only if you've decided the visual trade-off is worth it.

- **`buildPhishingBrief` ignores most of its parameters.** The function takes `(c, arts, wo, lk, poster, qr, illus)` but consumes only `arts`. Other phishing-brief siblings inherit this. Do not introduce dependence on those args without checking — they may be `undefined` for some callers.
